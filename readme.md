# AI 工程项目集（ai-public）

> **5 个从市场真实需求中长出来的可运行系统**
> **覆盖：AI 内容生成 · 订单风控 · 大模型微调 · 日报自动化 · AI-first 升级路径**

---

## 写在前面

今天，几乎每一家企业都在说"我们要上 AI"。

但真正落地时遇到的，不是 PPT 上的蓝图，而是这些具体的问题：

- "跨境电商要做 9 个语种的营销文案，现在靠翻译公司，**一周出 30 条、还要再花 3 天校对**——AI 能做到今天下午就出 300 条吗？"
- "每天 8000 单的交易，**漏掉 1 单欺诈就是几万块的损失**——AI 能不能从我现有的数据里学出来什么？"
- "客服那边想要一个能用 ChatGPT 的小工具，**但只懂 LLM 是不够的**——企业文档怎么喂给模型、怎么保证它不胡说八道？"
- "我团队 15 个 Java 工程师还在用老套路写 CRUD，**需求响应速度跟不上业务了**——AI 编码工具到底是噱头还是真的能帮上忙？"
- "开发日志我自己都记不清哪一天干了什么，**让 AI 帮我把每天做的事自动整理成日报**行不行？"

这些不是假想的需求。这些是我看到的真实场景——来自企业内部的吐槽、岗位 JD 的描述、以及行业里被反复验证过的痛点。

**这个仓库里的 5 个项目，每一个都对应上述其中一类问题**。它们不是"为了演示 AI 而演示 AI"的玩具，而是**从企业真实业务流程里长出来、能跑、能复用、能演进的工程实现**。

## 二、为什么是这 5 个项目，而不是 5 个玩具

不是每一个 AI Demo 都有意义。这 5 个项目有一个共同的筛选标准：

### ✅ 它们都满足三个条件

**1. 真实场景驱动**
每个项目的起点都是一个具体的、可量化的问题——比如"出 9 语种文案需要 1 周"——而不是"我想玩玩 LLM"。

**2. 完整工程闭环**
从数据输入，到模型/规则处理，到产出可消费的输出（HTML 看板、JSON 报告、文档），每个项目都跑通了端到端流程，不只是停留在 Notebook 里。

**3. 可被复用**
代码结构、Prompt 模板、Skill 定义、规约体系——这些都是可以平移到下一个项目或团队里去用的资产，不是用完就丢的一次性脚本。

## 三、5 个项目对应 5 类企业 AI 需求

### 🎯 需求 1：传统 Java 系统如何升级到 AI 协作模式？

> **对应项目：AIBuilder — AI 全栈开发工程师能力展示系统**

**场景画像**：
某家公司的 Java 后台已经运行 8 年，Spring Cloud 微服务架构，业务复杂但代码老旧。业务方提一个需求，从分析到上线要 2 周。CTO 听说了 AI 编码工具，但不知道在不重写的前提下，怎么让团队真的用起来。

**这个项目回答的问题**：
- 一套 8 微服务的 CMS 系统，**47 小时、18 个里程碑、用 AI 协作完成**，是可能的
- AI 编码工具不是"自动写代码"——它需要**Agent 角色分工、Skill 沉淀、6 道质量门禁、陷阱档案**这些配套机制
- 升级路径应该是**渐进式**的：先小团队试点（2-3x 效率），再沉淀规约（团队协作），再重构关键模块（可维护性），最后建立企业级 AI-first 工程能力

**对受众的意义**：
如果你是企业的技术负责人，你看到的不只是一个"会写代码的人"，而是一个**能把 AI 工具变成团队能力**的人——他懂得怎么避免"AI 写出一堆烂代码"这种最常见的失败模式。

[→ 详细了解](wiki/AIBuilder—AI全栈开发工程师能力展示系统.md)

### 🎯 需求 2：跨境电商的运营工作能否被 AI 流水线接管？

> **对应项目：EKS — 跨境电商 AI 运营平台**

**场景画像**：
一家做跨境电商的小公司，每天要处理 9 个语种的产品文案、客服咨询、订单审核、选品分析、运营复盘。人工成本高、效率有上限、错误率难控。

**这个项目回答的问题**：
- **多语种文案**：9 语种（中日韩英法德西葡阿）、3 种输出格式（TXT/MD/HTML）、按产品×语种矩阵批量生成——**这不再是周级工作，是小时级**
- **智能客服**：上下文感知的多轮对话，可作为真实客服的辅助层
- **订单风控**：PyTorch 3 层神经网络 + 8 维特征工程 + 时间平移机制——F1=1.0 的轻量模型，21KB，可以直接部署
- **选品分析 & 运营复盘**：从 CSV 到 3 格式报告（终端/知识库/浏览器），全程可视化

**对受众的意义**：
一家小公司不需要花 100 万去建中台，**一个人 + 一套开源工具 + 一组 API 就能搭起 80% 的 AI 运营能力**。这个项目就是可复制的样板。

[→ 详细了解](wiki/EKS-跨境电商AI运营平台介绍.md)

### 🎯 需求 3：AI 订单风控到底是怎么"工作"的？

> **对应项目：订单风控 PyTorch 神经网络原理**

**场景画像**：
上面那个跨境电商项目里，订单风控模块是黑盒——产品经理、运营、老板都会问："AI 到底怎么判断一笔订单是欺诈的？凭什么相信它？"

**这个项目回答的问题**：
- **特征工程**：8 个维度的设计逻辑——金额、注册天数、改址次数、30 天订单数、30 天均价、金额偏离比、下单时段、距今天数。每个特征为什么有效、对应什么业务假设
- **时间平移机制**：演示数据是"过去"的时间戳，但风控模型要在"今天"可用——这个项目用一段 30 行的 Python 代码解决了"训练数据和推理数据时间不一致"这个 90% 的 AI 项目都会踩的坑
- **模型结构**：3 层全连接 + BatchNorm + Dropout + Sigmoid——选 PyTorch 而不是 sklearn 的理由是"能展示深度学习落地能力"而不是为了炫技
- **风险看板**：Top20 高风险订单、时段分布、欺诈 vs 正常维度对比、4 个典型 Case 的实时推理——业务方一眼能看懂

**对受众的意义**：
这是一个**用真实工程语言解释 AI 决策过程**的范例——不是"神经网络玄学"，而是"特征工程 + 模型 + 业务判断 + 可视化"完整闭环。可以作为企业内部技术分享、培训新人的标准材料。

[→ 详细了解](wiki/EKS-跨境电商AI运营平台订单风控PyTorch神经网络原理.md)

### 🎯 需求 4：企业内部搭"微调 + RAG"知识库，到底应该怎么开始？

> **对应项目：LMFT — AI 大模型训练工程师全链路微调工作台**

**场景画像**：
某企业想用大模型做内部知识库——客服 SOP、产品手册、HR 政策都想喂给 AI。但只调 LLM 是不够的：怎么准备数据？怎么训练？怎么评估效果？微调完了发现模型还在胡说八道怎么办？

**这个项目回答的问题**：
- **5 个场景全链路**：数据工程 → LoRA 微调 → 评估对比 → 蒸馏量化 → RAG 融合——从"原始种子"到"可评估的微调模型"全跑通
- **3 层检验体系**：ROUGE-L / BLEU 自动评分 + 逐题对比 + 人工质检——不是只跑一个数字，而是能从数据里看出模型到底进步了哪里、退步了哪里
- **阶段版本快照**：每轮训练都存档（种子/数据/适配器/Loss 曲线/超参数），可对比可回滚——这是企业级生产环境才有的工程规范
- **微调 vs RAG 的路线判断**：经过三轮实际训练验证得出的硬核结论——"纯微调不能单独用于知识密集型问答，RAG 才是工业级正确的架构"

**对受众的意义**：
这个项目最珍贵的不是"我能跑通微调"，而是**"我能告诉你什么方案行不通、为什么、正确路应该怎么走"**——这在 AI 泡沫期是稀缺能力。20 年的工程经验沉淀出的是判断力，不是炫技。

[→ 详细了解](wiki/LMFT-AI大模型训练工程师全链路微调工作台介绍.md)

### 🎯 需求 5：AI 能不能帮我把"每天做了什么"自动整理出来？

> **对应项目：NoteBI — 日报自动生成系统**

**场景画像**：
一个 AI 重度使用者，每天在多个项目里切来切去（代码、文档、知识库、笔记），到晚上根本想不起来自己今天到底干了什么。手动写日报？不存在的，太费精力了。AI 能不能从我的"行为痕迹"里反推出工作内容？

**这个项目回答的问题**：
- **自动扫描文件变更**：从 PARA vault + 跨工作空间项目（eks-demo 等）扫描新建/修改的文件
- **行为模式识别**：自动识别峰值日、周末行为、环比变化、热点目录
- **7 天趋势折线图**：日报本身可以累积成周报，趋势清晰可见
- **结构化输出**：今日速览 → 输出文件 → 今日重点 → 分布统计 → 思维洞察 → 小规律 → 提醒建议
- **红线规则**：只读文件元数据（不读内容），绝不修改用户文件——这是对个人工作空间的尊重

**对受众的意义**：
你不需要一个花哨的日报系统，你只需要一个**不会打断你、不会污染你数据、能在你需要的时候给你看"你这周/这个月干了什么"** 的工具。

[→ 详细了解](wiki/NoteBI-日报自动生成系统介绍.md)
